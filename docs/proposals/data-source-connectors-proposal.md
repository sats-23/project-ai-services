# Remote connectors implementation plan

## Top-Level Overview

Introduce first-class connector support as a user-facing capability in the catalog product surface so users can create and manage remote data sources from the catalog UI and attach chosen connectors to a digital assistant application during application creation.

Recommended implementation direction:
- Catalog exposes the connector management APIs used by the UI.
- Catalog and Caddy jointly handle connector auth flows, callback routing, and preflight validation.
- Connector integrations are treated as a dedicated subsystem that normalizes provider-specific access to remote file systems, S3-like object stores, Google Drive, SharePoint, and similar sources.
- Connectors are shared catalog resources that multiple applications can attach to.
- Each digital assistant application deploys its own digitize instance.
- That per-application digitize instance consumes the shared connectors attached to its application, but does not participate in provider auth or provider health validation.
- Application creation must support attaching one or more existing shared connector resources to the created application instance.
- The implementation plan focuses on actual service boundaries, API additions, persistence, deployment, and runtime wiring. It does not include a test-planning workstream.

**SSH/SFTP connector scope:** The first remote file system connector type to be fully implemented is an SSH/SFTP connector targeting Unix/Linux remote machines. The system generates an Ed25519 key pair per connector at creation time, stores only the private key (encrypted at rest), displays the public key to the user once so they can add it to their remote server's `authorized_keys`, and subsequently connects over SFTP to enumerate and ingest files. Change detection uses a per-file SHA-256 checksum registry so only new or modified files are forwarded to the digitize service. No password or user-supplied private key is ever accepted — catalog-generated public-key authentication is the only supported method for this connector type.

This direction matches the current repo structure where authenticated public APIs already live in [`ai-services/internal/pkg/catalog/apiserver/router.go`](ai-services/internal/pkg/catalog/apiserver/router.go), application creation is centralized in [`ai-services/internal/pkg/catalog/apiserver/handlers/application_handler.go`](ai-services/internal/pkg/catalog/apiserver/handlers/application_handler.go) and [`ai-services/internal/pkg/catalog/apiserver/repository/application_service.go`](ai-services/internal/pkg/catalog/apiserver/repository/application_service.go), persistence models live in [`ai-services/internal/pkg/catalog/db/models`](ai-services/internal/pkg/catalog/db/models), migrations live in [`ai-services/internal/pkg/catalog/db/migrations/assets`](ai-services/internal/pkg/catalog/db/migrations/assets), and routable deployed services already integrate through [`ai-services/internal/pkg/proxy/caddy.go`](ai-services/internal/pkg/proxy/caddy.go) and [`ai-services/internal/pkg/catalog/apiserver/services/deployment/repository/podman/deployer.go`](ai-services/internal/pkg/catalog/apiserver/services/deployment/repository/podman/deployer.go). The `github.com/pkg/sftp` library is already an indirect dependency in [`ai-services/go.mod`](ai-services/go.mod) and `golang.org/x/crypto` is a direct dependency, so SSH key generation and SFTP transport require no new external dependencies.

## SSH/SFTP Connector — User-Facing Parameters

When a user creates an SSH/SFTP connector the following inputs are involved:

| Field | Direction | Description |
|---|---|---|
| **Public Key** | System → User (display only) | Ed25519 public key generated by catalog at connector creation. User copies this into `~/.ssh/authorized_keys` on the remote host. Never editable or re-submitted by the user. |
| **Host / IP Address** | User → System | The FQDN or IP of the remote server (e.g. `sftp.company.com` or `192.168.1.50`). |
| **Username** | User → System | The Unix account on the remote server the system will authenticate as. |
| **Remote Folder Path** | User → System | The absolute directory path on the remote server containing the files to ingest (e.g. `/var/www/documents/`). |

Port defaults to 22. No password field is exposed.

## SSH/SFTP Connector — Runtime Flow

```
Create Connector (POST /api/v1/connectors)
  → Catalog generates Ed25519 key pair
  → Stores encrypted private key in connector record
  → Returns public key to UI (displayed once, stored in DB for re-display)
  → Connector status: auth-required

User adds public key to remote server authorized_keys
  ↓

Validate Connector (POST /api/v1/connectors/{id}/validate)
  → ConnectorAuthService.ValidateSSH()
      → Dial host:22 with generated private key and provided username
      → Open SFTP session
      → Stat remote folder path
  → On success: connector status → ready
  → On failure: connector status → degraded + validation error stored

Application creation attaches ready connector
  ↓

ConnectorSyncService (background goroutine, per-connector ticker)
  → SSHConnector.Scan(remoteFolderPath)
      → SFTP ReadDir (recursive)
      → For each file: compute remote SHA-256 checksum
      → Compare against ConnectorFileChecksum registry in DB
      → New file     → queue for ingest
      → Modified file → queue for re-ingest
      → Deleted file  → mark as removed, notify digitize
  → For each queued file: stream file bytes over SFTP → send to digitize HTTP ingest API
  → Update ConnectorFileChecksum registry with new checksums and timestamps
  → Update connector last_synced_at, sync_status in DB
```

### 3. Implement SSH key generation, connector auth service, and SFTP preflight validation
- **Intent** — On connector creation, catalog generates an Ed25519 key pair, persists the encrypted private key, and exposes the public key. On validation, catalog opens a real SFTP connection using the generated key to verify reachability, credentials, and remote path access before the connector can be attached to an application.
- **Expected Outcomes** — Every SSH/SFTP connector is provisioned with a system-generated Ed25519 key pair at creation time. The catalog can perform a live SFTP preflight check on demand. No password or user-supplied key material is ever stored or accepted.
- **Todo List**
  1. Create `ai-services/internal/pkg/catalog/apiserver/services/connectors/` as a new package. Add `ssh_keygen.go` with a `GenerateEd25519KeyPair()` function that uses `golang.org/x/crypto/ssh` (`ssh.NewSignerFromKey`, `ed25519.GenerateKey`) to produce an Ed25519 private key and return the private key (PEM-encoded) and the public key in OpenSSH authorized-keys format (`ssh.MarshalAuthorizedKey`). Use `crypto/rand` as the source of entropy, consistent with the pattern in [`ai-services/internal/pkg/catalog/apiserver/services/auth/service.go`](ai-services/internal/pkg/catalog/apiserver/services/auth/service.go).
  2. Encrypt the PEM-encoded private key before writing it to the database. Use a symmetric encryption approach (e.g. AES-GCM with a catalog-configured secret) so that the stored value cannot be used without the running catalog service. Add a `EncryptPrivateKey` / `DecryptPrivateKey` pair to a `crypto.go` file in the same connectors package.
  3. Add a `ConnectorAuthService` interface in the connectors package with methods: `ProvisionKeys(ctx) (publicKeyOpenSSH string, encryptedPrivKeyPEM string, error)` and `ValidateSSHConnector(ctx, connector Connector) error`.
  4. Implement `ValidateSSHConnector`: decrypt the stored private key, parse it with `ssh.ParsePrivateKey` (from `golang.org/x/crypto/ssh`), build a `ssh.ClientConfig` using `ssh.PublicKeys` auth method and `InsecureIgnoreHostKey()` for the initial implementation (host key verification can be added later), dial `host:port` using `golang.org/x/crypto/ssh`, open an SFTP client session using `github.com/pkg/sftp`, call `sftpClient.Stat(remotePath)` to confirm the path exists and is accessible, then close cleanly.
  5. Call `ConnectorAuthService.ProvisionKeys()` inside the `CreateConnector` handler before persisting the record, so the generated keys are written as part of the same create operation.
  6. Call `ConnectorAuthService.ValidateSSHConnector()` from the background validation job triggered by the `ValidateConnector` API endpoint, updating the connector status to `ready` or `degraded` with the error message as appropriate.
  7. Keep the `ConnectorAuthService` interface provider-agnostic at the boundary — the SSH/SFTP implementation is one concrete type. Future providers (OAuth, S3 keys) implement the same interface pattern without changing the handler or the validation job.
- **Relevant Context** — [`ai-services/internal/pkg/catalog/apiserver/services/auth/service.go`](ai-services/internal/pkg/catalog/apiserver/services/auth/service.go), [`ai-services/internal/pkg/utils/cert.go`](ai-services/internal/pkg/utils/cert.go), [`ai-services/go.mod`](ai-services/go.mod) (`golang.org/x/crypto v0.52.0`, `github.com/pkg/sftp v1.13.9`)
- **Status** — [ ] pending

### 4. Implement the SSH/SFTP file scanner and checksum-based change detection
- **Intent** — Add the runtime file scanning logic that connects to a remote SSH/SFTP server, enumerates files under the configured remote path, computes SHA-256 checksums, and identifies new or modified files that must be ingested or re-ingested by the digitize service.
- **Expected Outcomes** — A `ConnectorScanner` component can open an SFTP session to a ready connector's remote server, walk the configured directory recursively, compare each file's checksum against the stored registry, and return a diff of files to add, update, and remove.
- **Todo List**
  1. Add `sftp_scanner.go` in `ai-services/internal/pkg/catalog/apiserver/services/connectors/`. Define a `ScanResult` struct with three slices: `ToIngest []RemoteFile`, `ToReIngest []RemoteFile`, `ToRemove []RemoteFile`. Define `RemoteFile` with `Path string`, `Size int64`, `ModTime time.Time`, `Checksum string`.
  2. Implement `SSHConnector.Scan(ctx, connector Connector) (ScanResult, error)`: decrypt private key, open SFTP session (reuse the auth logic from sub-task 3), call `sftpClient.Walk(connector.RemotePath)` to enumerate all regular files recursively. For each file, read its content over SFTP and compute a SHA-256 checksum using `crypto/sha256`. Do not buffer entire files in memory — stream into the hasher and discard bytes to avoid memory pressure.
  3. After computing checksums for all files in the remote walk, load the full `ConnectorFileChecksum` registry from the database for that connector. Compare: files present remotely but absent in DB → `ToIngest`; files present in both but with different checksum → `ToReIngest`; files present in DB but absent remotely → `ToRemove`.
  4. The `Scan` method must return the result without side effects — it does not write to the checksum registry. The caller (sync service, sub-task 5) is responsible for updating registry entries and invoking the digitize service after processing.
  5. Keep SFTP session creation, teardown, and retry logic encapsulated inside the scanner so the sync service never holds raw SSH handles.
- **Relevant Context** — `ai-services/internal/pkg/catalog/apiserver/services/connectors/` (new package from sub-task 3), [`ai-services/internal/pkg/catalog/db/repository`](ai-services/internal/pkg/catalog/db/repository) (checksum repo from sub-task 1), [`ai-services/go.mod`](ai-services/go.mod) (`github.com/pkg/sftp v1.13.9`)
- **Status** — [ ] pending

### 5. Add the connector sync service and digitize ingest integration
- **Intent** — Introduce a background `ConnectorSyncService` that runs on a per-connector ticker, invokes the SFTP scanner, streams new/changed files to the digitize HTTP ingest API, and updates the checksum registry and sync status after each cycle. Deleted remote files are reported to digitize for removal from the index.
- **Expected Outcomes** — A running connector sync loop periodically scans each ready connector, forwards only changed files to the digitize service, updates the checksum registry, and writes sync status and `last_synced_at` back to the connector record.
- **Todo List**
  1. Create `ai-services/internal/pkg/catalog/apiserver/services/connectors/sync.go`. Define `ConnectorSyncService` with a `Start(ctx)` method that models the pattern from [`ai-services/internal/pkg/catalog/apiserver/services/sync/sync_service.go`](ai-services/internal/pkg/catalog/apiserver/services/sync/sync_service.go): a background goroutine with a ticker, a sync mutex to prevent overlapping runs, and graceful shutdown via context cancellation.
  2. On each tick, `ConnectorSyncService` fetches all connectors in `ready` status from the repository, and for each one calls `SSHConnector.Scan()` from sub-task 4.
  3. For each file in `ScanResult.ToIngest` and `ScanResult.ToReIngest`: open the file over SFTP, stream its bytes to the digitize service's ingest API using the existing HTTP integration pattern in [`ai-services/internal/pkg/application/common/backup/digitize.go`](ai-services/internal/pkg/application/common/backup/digitize.go). Use the application's digitize endpoint URL, obtained by resolving the connector's attached applications and their deployed digitize instances.
  4. For each file in `ScanResult.ToRemove`: call the digitize service's delete/remove API (to be confirmed against the digitize API surface) and delete the corresponding `ConnectorFileChecksum` record.
  5. After successfully processing a file, upsert the `ConnectorFileChecksum` record (path, checksum, size, `last_seen_at`, `ingested_at`) using `ConnectorFileChecksumRepository.UpsertChecksum`.
  6. After completing a full scan cycle for a connector, update `connector.last_synced_at` and `connector.sync_status` (`ok` or `partial_error`). If the SFTP session itself cannot be opened, set `connector.status` to `degraded`.
  7. The sync interval per connector should be configurable (defaulting to a reasonable value such as 5 minutes) and stored in the connector record or a catalog-level config so it can be tuned without redeployment.
- **Relevant Context** — [`ai-services/internal/pkg/catalog/apiserver/services/sync/sync_service.go`](ai-services/internal/pkg/catalog/apiserver/services/sync/sync_service.go), [`ai-services/internal/pkg/application/common/backup/digitize.go`](ai-services/internal/pkg/application/common/backup/digitize.go), `ai-services/internal/pkg/catalog/apiserver/services/connectors/` (new package)
- **Status** — [ ] pending
